
# 🚀 AI Engineering Assistant  
### **Self-Hosted Internal AI Platform for GitHub, Confluence, Docs & Kubernetes**

A fully self-hosted AI platform that integrates with:

- GitHub repositories  
- Confluence pages  
- Local documentation files  
- Kubernetes clusters (RKE2 / K3s)  
- Internal LLMs (OpenWebUI)  

The system provides:

- Accurate answers based **only on your organization’s data**  
- Automatic documentation generation  
- Code explanation & analysis  
- Real-time Kubernetes troubleshooting  
- Secure internal knowledge assistant  
- Strictly offline operation  

All data stays inside your infrastructure — **no external calls** in production.

---

# 📌 Key Features

### 🔹 **1. Project-Based Workspace**
Each project contains its own sources:
- GitHub repos  
- Confluence spaces/pages  
- Local documentation files  
- Kubernetes clusters  

Each project gets:
- A dedicated Milvus vector DB collection  
- Completely isolated ingested knowledge  
- Project-level source configuration  

---

### 🔹 **2. Intelligent Data Ingestion**

#### **GitHub**
- Fetches repo contents using PAT  
- Extracts code, docs, configs  
- Splits text into manageable chunks  
- Embeds using **Qwen3-Embedding**  
- Stores vectors in Milvus  

#### **Confluence**
- Fetches pages via API  
- Cleans & converts HTML → text  
- Chunk → embed → store  

#### **Local Files**
Uploads:
- PDF  
- Markdown  
- TXT  
- YAML / JSON  
- Logs  
- Configs  

Extracts → chunks → embeds → stores.

#### **Kubernetes (optional metadata ingestion)**
- Deployments  
- Pod specs  
- ConfigMaps  
- Nodes  
- Events  

---

### 🔹 **3. Real-Time Kubernetes Intelligence**

Reads live cluster state via kubeconfig:
- List pods/nodes/namespaces  
- Fetch pod logs  
- Identify errors/warnings  
- Filter pods by image/tag/version  
- Restart count analysis  
- Live cluster diagnostics via LLM  

**Example:**  
“Which pods use Prometheus images?”  
→ System reads cluster → filters pods → LLM explains clearly.

---

### 🔹 **4. High-Accuracy RAG (Retrieval-Augmented Generation)**  
The system ensures **strictly data-grounded responses**:

1. Query → embed with Qwen3-Embedding  
2. Search Milvus (per-project collection)  
3. (Optional) query live Kubernetes data  
4. Build curated context  
5. Send context + question to your LLM  
6. LLM returns answer using ONLY this context  

✔️ No hallucinations  
✔️ No internet  
✔️ Data-safe and deterministic  

---

### 🔹 **5. OpenWebUI LLM Integration**
You can choose a model **per project** from available OpenWebUI models:

- Qwen3-Coder-30B (via OpenWebUI)  
- Any model exposed via **OpenWebUI API**  
- Qwen3-Embedding for embeddings

# 🧱 How It Works (Architecture)

## **1. Ingestion Pipeline**
```
source → extract text → chunk (300–500 tokens) → embed → Milvus
```
- Embeddings generated by Qwen3-Embedding  
- Vector metadata stored for traceability  

---

## **2. Query Processing**
```
user_query → embed → vector search → optional live K8s data → context → LLM
```

Ensures accurate, context-limited, data-grounded answers.

---

## **3. Backend**
- FastAPI service  
- Manages:
  - Projects  
  - Sources  
  - OpenWebUI integration (chat + embeddings)
  - Ingestion jobs  
  - Milvus operations  
  - Kubernetes queries  
  - RAG pipeline  
  - Chat sessions  

---

## **4. Frontend**
- Next.js (App Router)  
- React Query  
- TailwindCSS  
- Pages:
  - Projects  
  - Sources  
  - LLM Settings  
  - Chat  
  - Kubernetes explorer  

---

## **5. Self-Hosted Components**
All required components run locally:

- Milvus vector DB  
- Qwen3-Embedding  
- Qwen3-Coder-30B or any OpenWebUI model  
- Backend API  
- Frontend UI  
- Redis + Celery (optional ingestion queue)  

No external telemetry or network access.
